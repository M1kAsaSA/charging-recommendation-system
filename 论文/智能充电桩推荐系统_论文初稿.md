# 基于 Spring Boot 的智能充电桩推荐系统的设计与实现（论文初稿）

## 摘要
随着新能源汽车规模持续扩大，充电服务面临“找桩难、排队长、支付不便、管理滞后”等问题。本文围绕“基于 Spring Boot 的智能充电桩推荐系统”展开设计与实现，面向用户端微信小程序与管理端 Web 后台构建一体化充电服务平台。系统以多因子加权推荐为核心，融合距离、空闲率、电价与用户偏好进行综合评分，并提供地图展示、导航、扫码充电、订单结算与支付等功能；同时在运营侧提供充电站与设备档案管理、订单监控、异常处理与系统监控告警等能力。系统采用前后端分离架构，后端基于 Spring Boot + MyBatis 构建业务服务，前端管理端采用 Vue3 + Element Plus，小程序端采用微信原生语法，数据层使用 MySQL 与 Redis。本文给出需求分析、总体设计、详细设计与测试方案，形成了可部署、可扩展的智能充电推荐解决方案，为提升用户体验与运营效率提供了实践参考。

关键词：智能推荐；充电桩；微信小程序；Spring Boot；多因子加权

## ABSTRACT
With the rapid growth of electric vehicles, charging services still suffer from issues such as difficult station discovery, long queues, inconvenient payment, and inefficient operation management. This paper designs and implements an intelligent charging pile recommendation system based on Spring Boot, covering both a WeChat mini-program for end users and a web-based management console for operators. The system applies a multi-factor weighted recommendation strategy that combines distance, idle rate, price, and user preference to rank nearby stations. It also supports map display, navigation, QR-based charging, order settlement, and payment. On the operation side, it provides station and device management, order monitoring, exception handling, and system health monitoring. The system adopts a front-end/back-end separated architecture: the back-end is built with Spring Boot and MyBatis; the admin front-end is built with Vue3 and Element Plus; the mini-program uses native WeChat technologies. MySQL and Redis are used for data persistence and caching. This paper presents requirement analysis, overall architecture, detailed design, and testing plans, forming a deployable and extensible solution to improve user experience and operational efficiency.

KEY WORDS: intelligent recommendation; charging pile; WeChat mini-program; Spring Boot; multi-factor weighting

## 第1章 绪论
### 1.1 研究背景与意义
新能源汽车快速普及使充电基础设施成为关键配套。当前充电服务仍存在“可用性不可见、推荐不精准、运营管理分散”等问题。构建智能充电桩推荐系统能够在用户侧提升找桩效率与充电体验，在运营侧提升设备利用率与管理效率，具有明显的现实意义与应用价值。

### 1.2 智能充电桩管理与调度研究现状
国内外充电平台已实现地图检索、状态查询与支付等基础能力，但多数系统仍以单一指标驱动（如距离或价格），且运营侧与用户侧的数据联动不足。现有研究强调推荐算法优化、设备状态感知与负载均衡，但在多端一体化与实时监控方面仍有提升空间。

### 1.3 充电桩个性化推荐技术现状
已有方案主要围绕协同过滤、地理位置建模与负载预测展开，能在一定程度上实现个性化推荐。但由于实时空闲率、电价波动与用户偏好变化复杂，推荐准确性和可解释性仍需改进。多因子加权策略结合用户行为与实时数据，是当前工程落地较稳健的选择。

### 1.4 研究内容与技术路线
本文围绕“用户端推荐 + 管理端运维 + 系统监控”三类能力展开，技术路线如下：
1) 需求梳理与角色分析，明确用户端、管理端与监控端功能范围。
2) 设计前后端分离架构，完成模块划分与数据流梳理。
3) 建立核心数据模型与数据库结构。
4) 实现多因子推荐算法与关键业务流程（找桩、充电、订单与支付）。
5) 制定测试方案并进行功能与性能验证。

## 第2章 需求分析
### 2.1 系统应用场景与业务需求
本系统面向“用户找桩充电”与“运营方精细化管理”两类典型场景。用户侧的主要场景包括城市通勤途中临时补能、商圈/园区停车时段充电以及跨城出行途中快速充电等，特点是时效性强、地点不确定、对可用性与价格敏感。运营侧的主要场景包括站点与电桩的日常维护、运行状态同步、异常订单处置与经营统计分析等，要求数据实时、流程可追溯、管理可干预。

在上述场景下，系统需要满足以下核心业务需求：
- 附近充电资源可视化与快速检索：基于用户定位展示站点与电桩分布，支持按距离、空闲数、价格进行筛选与排序。
- 智能推荐与决策支持：综合距离、空闲率、电价与用户偏好给出推荐列表，帮助用户降低寻找成本并提升可达性。
- 一键导航与路径规划：支持从站点列表直接跳转至地图导航，降低操作复杂度。
- 充电全流程闭环：扫码启动、计费记录、结算支付、订单状态同步与历史订单查询，保证业务一致性与可追踪性。
- 站点与电桩全生命周期管理：支持站点与设备的新增、修改、下线与状态变更，形成可维护的设备档案。
- 运营与监控能力：订单监控、异常订单处理、设备运行参数与服务器资源监控，及时告警并辅助运维决策。

### 2.2 多端角色交互需求分析
系统角色包括普通用户、运营管理员与系统监控员，分别通过小程序与 Web 管理端访问系统，形成“用户服务-运营管理-运行监控”三层协同。

普通用户侧的交互关注低学习成本与高响应速度，核心交互流程为：授权登录与定位、地图展示与推荐列表、站点详情查看、扫码启动充电、订单查看与支付完成。用户在同一页面完成“搜索-选择-导航-充电-结算”闭环，减少跨页面操作。

运营管理员侧的交互关注设备状态与订单处理效率，核心交互流程为：站点与电桩档案维护、状态同步与手动干预、订单检索与异常处置、数据统计与运营分析。管理端以表格化和可视化为主，便于快速筛查异常与批量处理。

系统监控员侧的交互关注运行稳定性与告警响应，核心交互流程为：查看服务器资源指标与告警日志、识别异常阈值并追踪业务问题、与管理员协同处理设备故障或订单异常。监控端强调可视化指标与可追溯日志。

### 2.3 智能推荐与订单调度功能需求分析
推荐功能是用户侧核心能力，需满足“实时性、可解释性、可达性”三方面要求。系统在获取用户当前位置与搜索半径后，拉取周边站点的实时状态数据（空闲枪数、电价、站点拥挤度），并结合用户偏好（快充/慢充偏好、历史订单、收藏记录）进行多因子加权排序，输出推荐列表。推荐结果需支持刷新与动态更新，保证用户看到的是最新可用状态。

订单调度功能需保证流程一致性与异常可控。扫码后系统应校验电桩可用状态并生成订单，计费过程记录开始时间、电量与费用，结束后触发结算与支付。订单状态在“未支付-已支付-已完成/已取消”之间流转，并同步给管理端以便运营侧介入。对于长时间未支付、设备故障中断等场景，应提供超时关闭、人工干预与日志记录能力。

### 2.4 系统非功能需求分析
系统非功能需求主要包括性能、安全、可用性与可维护性四个方面。

性能方面，地图与推荐列表需在合理时间内返回结果，核心接口在并发场景下应保持稳定响应，避免因高并发造成服务阻塞。缓存策略用于降低热点数据访问压力。

安全方面，接口访问需进行身份认证与权限校验，支付与订单接口必须由后端签名生成关键参数；敏感数据如手机号、订单号等在日志中需做脱敏处理，并对用户信息进行必要的加密存储。

可用性方面，系统需在高峰期保持稳定运行，关键业务流程采用事务与必要的状态校验保证一致性，异常情况下能快速回滚或恢复。

可维护性方面，系统应具备结构化日志与监控指标，便于定位问题；配置项与运行参数可集中管理，支持后续扩展与部署迁移。

## 第3章 系统总体设计
### 3.1 系统总体架构设计
系统采用前后端分离架构，按功能与数据边界划分为交互层、网关与接入层、应用服务层、数据层与第三方服务层。

交互层包含微信小程序与管理端 Web，主要承担 UI 展示与交互，不承载复杂业务逻辑；网关与接入层通过 Nginx 统一入口与转发策略，完成跨端复用与路径路由；应用服务层基于 Spring Boot 提供 REST API，承载推荐、订单、设备管理与监控等核心业务；数据层以 MySQL 存储用户、站点、设备与订单等核心数据，并采用 Redis 作为缓存与会话支撑；第三方服务层包括地图 SDK 与微信支付接口，提供定位、导航与支付能力。

架构遵循“客户端轻、服务端重”的原则，将规则、校验与状态流转集中在服务端，降低前端复杂度并保证一致性。对外访问统一使用 `/api/v1` 前缀，身份认证通过 JWT 实现，避免前端直接掌控敏感参数。整体架构保持单体服务拆分边界清晰，便于后续按业务域演进。

### 3.2 功能模块划分
系统功能模块按业务域划分如下，模块之间以接口与数据实体进行解耦：
- 用户模块：负责微信登录鉴权、用户基本信息维护、偏好记录与收藏管理，向推荐模块提供用户画像与行为特征。
- 站点与设备模块：负责充电站档案、充电桩档案与状态同步，支持人工干预（维修、下线）与状态变更审计。
- 推荐模块：对候选站点进行多因子加权评分，提供排序结果与推荐解释字段（距离、空闲率、电价与偏好贡献）。
- 订单模块：承接扫码建单、计费记录、订单状态流转与支付回调处理，确保订单与支付状态一致，支持异常订单处置。
- 监控模块：汇聚 Actuator 指标与自定义业务日志，形成资源与业务监控面板，支持告警阈值设置。
- 管理端统计模块：对订单、收入、设备状态进行统计分析，为运营决策提供数据支撑。

### 3.3 数据流与业务流程设计
系统主要业务流程分为“推荐找桩”“扫码充电”与“运营管理”三条主链路。

推荐找桩数据流如下：
1) 用户授权登录并获取定位，经小程序请求推荐接口；
2) 服务端查询附近站点与电桩状态，结合 Redis 缓存与数据库数据进行候选集构建；
3) 推荐模块对候选集进行多因子加权评分并排序，返回推荐列表与站点详情；
4) 用户选择站点后进入详情页并发起导航，完成“发现-筛选-选择”的流程闭环。

扫码充电数据流如下：
1) 用户扫码获取电桩标识，服务端校验电桩状态与用户权限；
2) 订单模块生成订单并记录计费开始时间，设备模块同步状态为“使用中”；
3) 充电结束后生成结算金额，后端发起微信支付下单并返回签名参数；
4) 支付回调确认后更新订单状态，写入支付结果与日志，同时回写电桩状态并同步给管理端。

运营管理数据流如下：
1) 管理员维护站点/电桩档案与状态，操作记录写入日志；
2) 管理员通过订单检索与监控面板处理异常订单；
3) 统计模块周期性汇总订单与收入数据，为运营分析提供支撑。

### 3.4 关键技术路线设计
关键技术路线从“架构-数据-安全-性能-运维”五个层面展开：
- 架构层：Spring Boot 构建服务骨架，模块化拆分业务边界，REST API 统一对外输出；管理端与小程序通过统一 API 前缀接入。
- 数据层：MySQL 负责核心业务数据存储，Redis 提供热点数据缓存与会话支持，降低查询延迟并提升并发能力。
- 安全层：JWT 鉴权与权限校验控制接口访问，支付与订单相关参数由后端签名生成，避免前端篡改。
- 性能层：推荐列表与站点状态引入缓存策略，关键查询接口采用分页与索引优化，降低响应时间。
- 运维层：Docker Compose 实现后端、管理端、数据库与缓存的统一部署，日志与监控指标接入 Actuator 与自定义日志体系。

## 第4章 系统详细设计
### 4.1 数据库设计与持久层实现
核心实体包括用户、充电站、充电桩、订单、收藏与监控日志。实体关系以用户与订单、站点与充电桩为主线，持久层采用 MyBatis 实现 CRUD 与分页查询。

### 4.2 基于用户偏好的智能推荐模块设计
推荐模块采用多因子加权策略，综合距离、空闲率、电价与偏好评分。

$$
Score = W_1 \cdot N(Distance) + W_2 \cdot N(Idle) + W_3 \cdot N(Price) + W_4 \cdot N(Preference)
$$

其中 $W_1=0.30$、$W_2=0.40$、$W_3=0.20$、$W_4=0.10$，各指标采用归一化处理。偏好分数来自用户历史订单、收藏站点与充电类型偏好。

### 4.3 移动端用户交互与导航模块设计
小程序端提供地图展示、站点列表与推荐卡片。用户可查看站点详情（地址、充电类型、空闲数、电价），并一键导航至目标站点。扫码充电流程由小程序调用后端接口启动。

### 4.4 站点与电桩状态监控模块设计
管理端提供站点列表与状态面板，系统定期同步设备心跳数据，支持标记故障、维修中与强制下线等操作。监控模块基于 Spring Boot Actuator 输出服务器资源指标并形成告警日志。

### 4.5 充电计费与订单业务流程设计
订单流程包括扫码建单、计费开始、计费结束、支付与状态回写。支付流程由后端对接微信支付，生成支付签名，避免前端直接拼接金额参数。异常订单支持手动干预（强制结束、作废等）。

### 4.6 系统集成与关键流程说明
系统通过统一 API 前缀管理服务入口，管理端通过 Nginx 反向代理访问后端服务，云端部署采用 Docker Compose 统一启动后端、管理端与数据库服务，实现一体化发布。
