### **1. 引言**

#### **1.1 编写目的**
本需求规格说明书旨在详细定义“基于Spring Boot的智能充电桩推荐系统”的功能性与非功能性需求。本文档的预期读者包括项目指导教师、系统开发人员（前后端）、测试人员及项目管理人员，作为后续系统设计、编码实现、测试验收及维护迭代的共同依据。

#### **1.2 产品范围**
本系统定位于为电动汽车用户提供“即用即走”的智能充电服务，同时为运营商提供全维度的数字化管理工具。系统覆盖**用户端微信小程序**与**管理后台Web端**两大终端，重点解决用户在寻桩过程中的“里程焦虑”与运营方的“管理滞后”问题。

#### **1.3 术语定义**
- **SRS**：软件需求规格说明书。
- **多因子加权推荐**：综合考虑距离、电价、空闲率、用户偏好等多个维度计算权重的推荐算法。
- **IOT**：物联网，本系统特指充电桩设备的联网状态与运行参数采集。

---

### **2. 总体描述**

#### **2.1 产品背景与愿景**
随着新能源汽车保有量的爆发式增长，充电基础设施面临“找不到、排队长、管理乱”的行业痛点。本系统旨在通过 **Spring Boot + 微信小程序** 技术栈，构建一个集 **精准推荐、全维管理、实时监控** 三位一体的智能化充电服务平台，提升用户充电体验与运营维护效率。

#### **2.2 用户角色分析**

| 角色                  | 主要职责与需求                                               | 使用终端               |
| :-------------------- | :----------------------------------------------------------- | :--------------------- |
| **普通用户 (EV车主)** | 快速找到可用充电桩、获取最优推荐、便捷支付、查看历史订单。   | 微信小程序             |
| **运营管理员**        | 管理充电桩设备档案、监控设备状态（正常/故障）、处理异常订单、查看运营数据。 | Web管理后台            |
| **系统监控员**        | 监控服务器负载、数据库连接数、业务异常告警，保障系统高可用。 | Web管理后台 (监控看板) |

#### **2.3 运行环境约束**
- **用户端**：微信 App (iOS/Android)，需支持定位权限与微信支付。
- **管理端**：Chrome 80+ 或 Edge 浏览器。
- **服务端**：Linux 服务器，Java 8/11 运行环境，MySQL 5.7+。

| **F-USER-01** | 地图展示     | 系统需集成腾讯/高德地图 SDK，默认展示以用户当前位置为中心的地图视图，并标记周边充电桩点位。 |

| :------------ | :------- | :----------------------------------------------------------- |

| **F-USER-06** | 订单管理 | 用户可查看“进行中”订单（实时电量、费用、时长）及“历史订单”详情。 |
| **F-USER-07** | 在线支付 | 充电结束自动生成账单，系统对接**微信支付 API**，用户确认金额后完成支付。 |

| **F-ADMIN-04** | 异常订单干预 | 针对长时间未支付挂起或结算异常的订单，提供“强制结束计费”、“补扣款”、“作废”等管理手段。 |

| :----------- | :--------------- | :----------------------------------------------------------- |

### **4. 非功能性需求**

#### **4.1 性能需求**
- **响应时间**：95% 的查询操作（如查询附近充电桩）响应时间应小于 **800ms**；地图加载时间小于 **2s**。
- **并发能力**：系统需支持高峰期至少 **200** 用户同时进行扫码请求不卡顿，数据库连接池配置合理，利用 Redis 缓存热点数据。

#### **4.2 安全性需求**
- **接口鉴权**：所有后端 API 必须经过 Token (JWT) 校验，防止未授权访问。
- **支付安全**：支付参数必须由后端签名生成，严禁前端直接拼接金额参数。
- **数据加密**：用户登录密码必须采用 BCrypt 或 MD5 加盐存储，敏感日志脱敏处理。

#### **4.3 可用性与可靠性**
- **服务可用性**：系统整体可用性目标为 **99.5%**（允许夜间短暂维护）。
- **数据一致性**：涉及充电计费与余额扣减的流程，需使用数据库事务或分布式锁保证数据最终一致性。

#### **4.4 可维护性**
- **日志规范**：系统需记录详细的操作日志和异常日志，便于故障排查。

### **6. 数据需求（逻辑模型示例）**

为满足上述业务，系统核心数据实体需包含以下关键字段：

| 实体名                     | 核心字段                                                     | 说明                             |
| :------------------------- | :----------------------------------------------------------- | :------------------------------- |
| **User (用户)**            | `user_id`, `openid`, `phone`, `status`                       | 关联微信小程序唯一标识。         |
| **ChargingStation (站点)** | `station_id`, `name`, `lng`, `lat`, `address`                | 充电站地理信息。                 |
| **ChargingPile (电桩)**    | `pile_id`, `station_id`, `type`, `power`, `price`, `status`  | 具体设备信息，状态由管理端维护。 |
| **Order (订单)**           | `order_no`, `user_id`, `pile_id`, `start_time`, `end_time`, `energy`, `amount`, `pay_status` | 核心业务流水表，支撑数据分析。   |
| **AlarmLog (告警日志)**    | `log_id`, `type`, `content`, `create_time`                   | 用于监控模块异常追溯。           |
